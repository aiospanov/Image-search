# PRD — Поиск товаров по изображению

**Версия:** 1.0 · **Статус:** Draft · **Дата:** Июнь 2025  
**Стек:** Python · PostgreSQL · Elasticsearch  
**Аудитория:** Команда разработки

---

## 1. Контекст и цель

Пользователи маркетплейса регулярно сталкиваются с задачей найти товар, который они видели «где-то в интернете» или сфотографировали в магазине, но не знают как его назвать. Текстовый поиск в таких случаях неэффективен — пользователь уходит без покупки.

Функция поиска по изображению решает эту проблему: пользователь загружает фото или делает снимок камерой, система анализирует изображение через Google Vision API, извлекает метки, переводит их через словарь синонимов и находит релевантные товары в каталоге через Elasticsearch.

### Проблема

- Пользователь не может сформулировать поисковый запрос словами
- Конверсия в покупку падает при отсутствии результатов по тексту
- Конкуренты (крупные площадки) уже имеют визуальный поиск

### Цель

- Добавить точку входа «поиск по фото» в интерфейс маркетплейса
- Обеспечить точность возврата релевантных результатов ≥ 70% (Precision@5)
- Запустить MVP за 4–6 недель силами команды

## 2. Целевые пользователи

### Покупатель с фото

- Нашёл товар офлайн или в соцсети, хочет купить онлайн
- Не знает точного названия или бренда
- Ожидает: загрузил фото → увидел похожие товары за 2–3 секунды

### Покупатель «по вдохновению»

- Видит товар у знакомых, в кино, на улице
- Делает фото камерой телефона прямо в приложении
- Ожидает: навёл камеру → нашёл в каталоге

## 3. Функциональные требования

Приоритеты: `Must` = обязательно для MVP · `Should` = желательно · `Could` = будущие версии

### 3.1 Загрузка изображения

| ID | Описание | Приоритет |
|----|----------|-----------|
| FR-01 | Пользователь может загрузить файл (jpg, png, webp) размером до 10 МБ с устройства | Must |
| FR-02 | Пользователь может сделать фото через камеру прямо в браузере/приложении | Must |
| FR-03 | Превью загруженного изображения отображается до отправки запроса | Should |
| FR-04 | Валидация формата и размера файла на фронтенде до отправки на сервер | Must |

### 3.2 Анализ изображения

| ID | Описание | Приоритет |
|----|----------|-----------|
| FR-05 | Изображение отправляется в Google Vision API, запрашивается только `LABEL_DETECTION` | Must |
| FR-06 | Возвращаются метки с confidence ≥ 0.70, берётся топ-5 | Must |
| FR-07 | Метки на английском маппируются через словарь синонимов на русские термины каталога | Must |
| FR-08 | Метки без словарного маппинга переводятся через Google Translate API как fallback | Should |
| FR-09 | Весь запрос (фото → метки → результаты) выполняется не дольше 3 секунд | Must |

### 3.3 Поиск по каталогу

| ID | Описание | Приоритет |
|----|----------|-----------|
| FR-10 | Из русских меток формируется запрос и выполняется `multi_match` в Elasticsearch по полям `name`, `description`, `category`, `brand` | Must |
| FR-11 | Поиск выполняется с `fuzziness: AUTO` для устойчивости к падежам и опечаткам | Must |
| FR-12 | Результаты дополнительно фильтруются по наличию товара на складе | Must |
| FR-13 | Возвращается не более 20 товаров, отсортированных по relevance score | Must |
| FR-14 | При 0 результатах показывается friendly-сообщение с предложением попробовать другое фото | Must |

### 3.4 Отображение результатов

| ID | Описание | Приоритет |
|----|----------|-----------|
| FR-15 | Результаты отображаются в той же сетке, что и обычный поиск (карточки товаров) | Must |
| FR-16 | Над результатами отображаются применённые метки как индикатор того, «что нашла система» | Should |
| FR-17 | Пользователь может нажать на тег и убрать его для уточнения поиска | Could |
| FR-18 | Кнопка «Искать по фото» доступна на главной странице и в строке поиска | Must |

## 4. Нефункциональные требования

### 4.1 Производительность

| ID | Описание | Приоритет |
|----|----------|-----------|
| NFR-01 | P95 latency полного запроса (загрузка → результаты) не превышает 3 с при нормальной нагрузке | Must |
| NFR-02 | Сервис выдерживает до 50 одновременных запросов без деградации | Must |
| NFR-03 | Изображение сжимается до 1 МБ перед отправкой в Vision API для снижения задержки | Should |

### 4.2 Надёжность

| ID | Описание | Приоритет |
|----|----------|-----------|
| NFR-04 | При недоступности Google Vision API возвращается понятная ошибка, функция не ломает остальной поиск | Must |
| NFR-05 | Retry-логика: при ошибке Vision API — 1 повтор через 500 мс | Should |
| NFR-06 | Uptime функции ≥ 99.5% в рабочие часы | Should |

### 4.3 Безопасность

| ID | Описание | Приоритет |
|----|----------|-----------|
| NFR-07 | Загруженные изображения не сохраняются на сервере после обработки запроса | Must |

| NFR-09 | Ограничение: не более 10 запросов с одного IP в минуту (rate limiting) | Should |

## 5. Техническое решение

### 5.1 Поток данных

```
Пользователь загружает фото
        ↓
Фронтенд: сжатие до 1 МБ → POST /api/search/image
        ↓
Python/FastAPI: валидация файла
        ↓
Google Vision API → метки EN (confidence ≥ 0.70, топ-5)
        ↓
label_mapper.py: словарь EN→RU + Google Translate fallback
        ↓
Elasticsearch: multi_match + fuzziness: AUTO → список SKU
        ↓
PostgreSQL: обогащение данными товара (цена, фото, наличие)
        ↓
JSON: {products: [...], applied_labels: [...]}
```

### 5.2 Новые компоненты

| Компонент | Назначение |
|-----------|------------|
| `POST /api/search/image` | Новый эндпоинт приёма фото |
| `vision_service.py` | Обёртка над Google Vision API |
| `label_mapper.py` | Словарь EN→RU + Translate fallback |
| `image_search.py` | Elasticsearch multi_match запрос |
| Таблица `label_synonyms` (PostgreSQL) | Хранение словаря меток |

### 5.3 Зависимости

- `google-cloud-vision` — анализ изображений
- `google-cloud-translate` — fallback-перевод
- `elasticsearch-py` — уже в стеке, новый индекс не нужен
- `Pillow` — сжатие изображений перед отправкой

## 6. Управление словарём меток

Словарь — ключевой компонент, влияющий на качество поиска.

### 6.1 Структура таблицы

```sql
CREATE TABLE label_synonyms (
    label_en   TEXT PRIMARY KEY,
    terms_ru   TEXT[],
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Пример записей
INSERT INTO label_synonyms VALUES
  ('sneakers',    ARRAY['кроссовки', 'кеды'],         NOW()),
  ('smartphone',  ARRAY['смартфон', 'телефон'],        NOW()),
  ('jacket',      ARRAY['куртка', 'жакет', 'ветровка'],NOW());
```

### 6.2 Стратегия наполнения

**До запуска:** собрать топ-50 категорий каталога → вручную составить маппинг. Покрывает ~80% типовых запросов.

**После запуска:** логировать метки без словарного маппинга. Раз в неделю PM + аналитик просматривают лог и добавляют новые записи.

**KPI:** доля запросов без словарного маппинга < 20% через 4 недели после запуска.

### 6.3 Кэширование

Словарь кэшируется в памяти приложения с TTL 1 час — исключает запрос к БД на каждый поиск.

| Этап | Описание | Срок | Артефакты |
|------|----------|------|-----------|

## 8. Оценка стоимости

Расчёт для маркетплейса с небольшим трафиком (до 10 тыс. поисков по фото в месяц):

| Статья | Объём/мес | Стоимость/мес | Примечание |
|--------|-----------|---------------|------------|
| Google Vision API | до 1 000 запросов | $0 | Первые 1 000 ед./мес бесплатно |
| Google Vision API | 1 001 – 10 000 | ~$13.5 | $1.50 за 1 000 ед. |
| Google Translate API (fallback) | ~20% от Vision-запросов | ~$1–2 | Только метки без словаря |
| **Итого при 10 000 поисков/мес** | — | **~$15/мес** | Растёт линейно с трафиком |
| Начальные кредиты Google Cloud | — | $300 (разовые) | Для новых аккаунтов GCP |

При типичном трафике небольшого маркетплейса затраты составят менее $20/месяц. Функция экономически оправдана даже при минимальной конверсии.

## 9. Метрики успеха

### Качество поиска

- **Precision@5 ≥ 70%** — не менее 3–4 из 5 результатов релевантны теме запроса
- Доля запросов с 0 результатами < 15%
- Доля запросов, обработанных словарём (без Translate fallback) ≥ 80% через 4 нед.

### Производительность

- P95 latency < 3 секунд
- Ошибки Vision API < 0.5% запросов

### Бизнес

- Используемость: ≥ 5% сессий содержат хотя бы 1 поиск по фото через 2 мес. после запуска
- Конверсия из поиска по фото в добавление в корзину — замеряем, цель определяем после 1-го месяца данных

---

## 10. Вне области видимости (MVP)

- Векторный поиск по эмбеддингам — рассматривается в v2 при росте каталога
- Поиск похожих товаров «найди визуально похожее» без текстового матча
- Распознавание текста и штрихкодов на фото
- Персонализация результатов на основе истории пользователя

## 11. Риски и митигация

| ID | Риск | Митигация |
|----|------|-----------|
| R-01 | Vision API возвращает нерелевантные метки → пустые результаты | Порог confidence 0.70, минимум 3 метки для запроса |
| R-02 | Словарь не покрывает специфику каталога (узкие категории) | Анализ топ-50 SKU до запуска, постоянное пополнение |
| R-03 | Google Vision недоступен | Graceful degradation — пользователь видит ошибку, основной поиск работает |
| R-04 | Пользователи загружают неподходящие фото (скриншоты, белый фон) | Инструкция в UI, friendly-ошибка при 0 меток |
| R-05 | Рост стоимости API при вирусном росте трафика | Alert при превышении 50 000 запросов/мес |
